# AI 交叉 Review 指南

## 什么是 AI 交叉 Review？

**原理**：用 AI A 写代码，用 AI B（或同一 AI 的新对话）来严格 review。

**为什么有效**：
- 新对话没有"我刚写了这段代码"的心理包袱
- 不同 AI 模型可能各有优势
- 避免"确认偏误"：同一对话的 AI 往往认可自己刚写的东西

---

## 交叉 Review Prompt 模板

### 模板 1: 完整功能 Review

```
你是一位资深 Python 代码审查员。请严格 review 以下代码，不要手下留情。

## 需求
[一句话描述功能目标]

## 代码变更
\`\`\`python
[粘贴代码或 git diff]
\`\`\`

## 请检查

1. **潜在 Bug**
   - 有没有边界情况没处理？
   - 有没有可能的 None/空值未保护？
   - 并发安全吗（如果涉及状态）？

2. **类型安全（mypy 视角）**
   - 类型注解完整准确吗？
   - 有没有 `Any` 类型滥用？

3. **测试充分性**
   - 哪些情况没有测试覆盖？
   - 现有测试有没有测错方向的？

4. **Python 规范（ruff 视角）**
   - 有没有 ruff 会抱怨的写法？
   - 有没有更 Pythonic 的实现？

5. **设计问题**
   - 职责单一吗？
   - 有没有明显的重构空间？

请以结构化的方式输出，区分"必须修复"和"建议改进"。
```

### 模板 2: 快速 Diff Review

```
请 review 这个 git diff，找出关键问题：

\`\`\`diff
[粘贴 git diff]
\`\`\`

重点关注：bug、缺失的类型注解、异常处理缺失、测试遗漏。
输出按优先级排序，最关键的问题在前。
```

### 模板 3: 测试 Review（只看测试）

```
请 review 这些 pytest 测试，判断它们是否充分：

\`\`\`python
[粘贴测试代码]
\`\`\`

被测功能描述：[简述]

检查：
1. 测试覆盖了哪些情况？
2. 遗漏了哪些重要情况？
3. 有没有测试之间的重复或矛盾？
4. fixtures 使用合理吗？
5. 测试隔离性好吗（互相不依赖）？
```

## 交叉 Review 操作流程

```
1. 完成功能开发（代码 + 测试）
         ↓
2. 运行质量链（ruff + mypy + pytest 全通过）
         ↓
3. 获取 diff
   git diff main..HEAD    （与主分支的差异）
   git show HEAD          （最近一次提交）
         ↓
4. 开新对话（或用不同模型）
         ↓
5. 粘贴 Prompt 模板 + diff
         ↓
6. 审阅 review 结果，决定哪些要修复
         ↓
7. 修复 → 再次运行质量链
         ↓
8. 提交
```

## 从 Review 中学习

记录每次交叉 review 发现的问题类型，建立个人 checklist：

| 常见问题 | 发现次数 | 预防方法 |
|---------|---------|---------|
| 未处理的 None | - | 所有函数都考虑 `X | None` |
| 可变默认参数 | - | 记住 `def f(x=None)` 模式 |
| 测试只覆盖 happy path | - | 写测试时先列 edge cases |
| 裸 except | - | 总是 `except SpecificError` |

## 常见发现的问题类型

基于实际经验，AI 交叉 review 经常能发现：

**Bug 类：**
- `list.sort()` 返回 `None`，赋值给变量后用了错误的值
- 字典 `.get(key)` 返回 `None` 但后续代码假设有值
- 字符串 `.strip()` 后忘记重新赋值

**类型类：**
- 函数可以返回 `None` 但返回类型写成了非 Optional
- 列表推导中类型被推断为 `list[Any]`

**测试类：**
- 只测了正常路径，没有测试第三方服务挂掉的情况
- Mock 了太多，实际上没有测到任何真实逻辑
- 测试依赖全局状态，单独运行可能失败

**设计类：**
- 一个函数做了两件事，应该拆分
- 硬编码了不应该硬编码的值
